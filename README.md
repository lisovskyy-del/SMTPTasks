# SMTPTasks

## Завдання 1: Дослідження SMTP протоколу «вручну» через telnet

**Що таке SMTP і на якому транспортному протоколі (TCP/UDP) він працює?**

SMTP - це протокол при якому ми можемо передавати листи іншим поштовим скринькам. SMTP працює на протоколі TCP, це необхідно так як листи мають дійти в цілісності.

**Яка різниця між командами HELO і EHLO?**

HELO - це застаріла версія привітання сервера від клієнта.

EHLO - (Extended Hello) вже сучасна версія привітання в ESMTP

**Навіщо потрібна крапка . на окремому рядку наприкінці повідомлення?**

Це ознака кінця повідомлення (end-of-data marker).

**На яких портах зазвичай працює SMTP? Чим відрізняються порти 25, 465, 587?**

SMTP працює на портах 25, 465, 587

25 - SMTP Relay. Сервер -> Сервер

465 - SMTPS (Legacy) - Застаріла схема.

587 - STARTTLS. Клієнт -> Сервер

## Завдання 2: SMTP-коди відповідей та команди

**1. Доповніть таблицю основних SMTP-команд:**

| Команда   | Призначення                                      | Приклад                          |
| --------- | ------------------------------------------------ | -------------------------------- |
| EHLO/HELO | Привітання та ідентифікація клієнта              | `EHLO mail.example.com`          |
| MAIL FROM | Вказати адресу відправника (конверт)             | `MAIL FROM:<user@example.com>`   |
| RCPT TO   | Вказати адресу отримувача (конверт)              | `RCPT TO:<admin@example.com>`    |
| DATA      | Почати передачу тіла листа                       | `DATA`                           |
| QUIT      | Завершити з'єднання                              | `QUIT`                           |
| AUTH      | Аутентифікація клієнта                           | `AUTH PLAIN AHVzZXJAZXhhbXBsZS5jb20AcGFzc3dvcmQ=` |
| STARTTLS  | Ініціює перехід від plaintext до TLS             | `STARTTLS`                       |
| RSET      | Припиняє передачу                                | `RSET`                           |
| VRFY      | Перевірка дійсності адреси                       | `VRFY user@domain.com`           |

**2. Доповніть таблицю кодів відповідей SMTP:**

| Код | Значення                      | Коли зустрічається                           |
| --- | ----------------------------- | -------------------------------------------- |
| 220 | Сервіс готовий                | При підключенні до сервера                   |
| 221 | Сервіс завершує роботу        | У відповідь на QUIT                          |
| 250 | Запитана дія виконана         | Після EHLO, MAIL FROM, RCPT TO               |
| 354 | Починайте передачу листа      | У відповідь на DATA                          |
| 421 | Сервіс недоступний            | Перевантаження                               |
| 450 | Поштова скринька недоступна   | Лист не вдалося доставити                    |
| 500 | Синтаксична помилка команди   | Коли команда не можна обробитись             |
| 550 | Дія не виконана, скринька н/д | Постійна помилка - не повторювати            |
| 552 | Ліміт розміру                 | Лист перевищує SIZE сервера                  |

**Чим відрізняються коди 4xx від 5xx у SMTP?**

4xx зустрічаються коли стається ТИМЧАСОВА помилка. Треба просто спробувати ще раз.

5хх зустрічаються коли вже стається постійна помилка. Треба виправити лист

**Що означає код 354 і чому він відрізняється від 250?**

354 означає проміжний. Він чекає дані листа. Він ще нічого не зробив, лиш чекає

250 означає те що він прийняв команду.

**Що станеться, якщо сервер поверне 550 у відповідь на RCPT TO?**

Якщо станеться 550 при RCPT TO, то сесія не переривається. SMTP продовжується далі.

## Завдання 3: Структура email-повідомлення (RFC 5322)

**1. Дослідіть структуру листа:**

```
From: Іван Петренко <ivan@example.com>
To: Марія Коваль <maria@example.com>
Cc: boss@example.com
Subject: =?UTF-8?B?0KLQtdGB0YLQvtCy0LUg0LfQsNCy0LTQsNC90L3Rjw==?=
Date: Sun, 24 May 2026 12:00:00 +0300
Message-ID: <unique-id@example.com>
MIME-Version: 1.0
Content-Type: multipart/mixed; boundary="----=_Part_1"

------=_Part_1
Content-Type: text/plain; charset=UTF-8

Привіт! Це тестовий лист.

------=_Part_1
Content-Type: application/pdf; name="report.pdf"
Content-Transfer-Encoding: base64
Content-Disposition: attachment; filename="report.pdf"

JVBERi0xLjQK...
------=_Part_1--
```

**2. Доповніть таблицю заголовків листа:**

| Заголовок        | Призначення                               | Обов'язковий? |
| ---------------- | ----------------------------------------- | ------------- |
| From             | Адреса відправника                        | Так           |
| To               | Адреса(и) отримувача                      | Так           |
| Subject          | Тема листа                                | Ні            |
| Date             | Дата та час відправки                     | Так           |
| Message-ID       | Унікальний ідентифікатор листа            | Ні            |
| Cc               | Копія                                     | Ні            |
| Bcc              | Прихована копія                           | Ні            |
| Reply-To         | Адреса для відповіді                      | Ні            |
| MIME-Version     | Оголошує, що лист використовує MIME       | Так           |
| Content-Type     | Тип вмісту листа                          | Ні            |


**Що таке MIME і навіщо він потрібен для email?**

MIME - це стандарт форматування вмісту листа. Замість того щоб передавати лист лише в 7-bit ASCII тексті, MIME дозволяє передаваті UTF-8 текст, HTML, вкладені файли, зображення, аудіо

**Чим відрізняються To, Cc та Bcc на рівні SMTP-протоколу?**

To - основні одержувачі листа

Cc - куда йде копія листа

Bcc - куда йде прихована копія листа.

**Що означає Content-Transfer-Encoding: base64 і чому вкладення кодуються у base64?**

Content-Transfer-Encoding: base64 - це кодування передачі вмісту на Base64. Це було зроблено для гарантування безпечної передачі бінарних даних.

**Чи може адреса у From заголовку відрізнятися від адреси у MAIL FROM? Які наслідки?**

Так, може. Це два різних поля. 

MAIL FROM - адреса на конверті (бачить лише пошта)

From - підпис всередині листа (бачить отримувач)
